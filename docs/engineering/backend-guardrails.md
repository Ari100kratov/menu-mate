# Правила backend-разработки

Эти правила обязательны для backend-работы в MenuMate. Их задача — не дать будущим изменениям незаметно ухудшить тестируемость, безопасность и сопровождаемость.

## Структура

- Доменные проекты должны быть разложены по каталогам `Models`, `ValueObjects`, `Enums`, `Errors`, `Services`, если в модуле есть соответствующие типы.
- Persistence-типы инфраструктуры должны жить в `Database/Entities`.
- EF Core конфигурации persistence-сущностей должны жить в `Database/Configurations` и подключаться через `ApplyConfigurationsFromAssembly`.
- Enum-поля в persistence-типах должны быть enum-типами в C#, а строковое хранение в БД настраивается через `HasConversion<string>()`; `Enum.Parse` для маппинга persistence-сущностей запрещен.
- DbContext не должен содержать разросшиеся inline-конфигурации сущностей.
- Все XML summary и markdown-документация должны быть на русском языке.

## Идентификаторы и связи между модулями

- Для persisted ID используем `Guid.CreateVersion7()`.
- Для persisted Guid v7, которые создаются приложением, EF Core конфигурация должна явно задавать `ValueGeneratedNever()` для ключевого свойства.
- `Guid.NewGuid()` нельзя использовать для идентификаторов, которые сохраняются в БД.
- Если поле обозначает связь с сущностью из другого модуля или другой БД, оно должно быть явным типом из `SharedKernel.Identifiers`, например `UserId` или `RecipeId`.
- Не добавлять новые cross-module поля вида `OwnerUserId: Guid`, `RecipeId: Guid?` в доменную модель.

## Время

- Нельзя напрямую вызывать `DateTimeOffset.UtcNow`, `DateTime.UtcNow`, `DateTime.Now` или `DateTimeOffset.Now` в прикладном, инфраструктурном и presentation-слое.
- Время приходит через `TimeProvider`.
- Тесты используют фиксированное время или fake `TimeProvider`.

## Качество сборки

- Перед возвратом работы запускать:
  - `dotnet restore MenuMate.slnx`
  - `dotnet build MenuMate.slnx --no-restore`
  - `dotnet test MenuMate.slnx --no-build`
- `TreatWarningsAsErrors`, `CodeAnalysisTreatWarningsAsErrors` и `EnforceCodeStyleInBuild` должны оставаться включенными.
- Подсказки анализаторов, влияющие на читаемость или корректность, нужно исправлять.
- Подавления анализаторов допустимы только точечно и с понятным обоснованием.
- Новые подавления анализаторов нельзя добавлять без явного подтверждения владельца проекта.
- Для EF Core миграций не добавляем новые подавления: сгенерированный код нужно привести к текущим analyzer-правилам вручную или пересоздать так, чтобы сборка оставалась чистой.

## HTTP и API

- Ошибки прикладного и доменного уровня возвращаются в формате проблем RFC 7807.
- 401 и 403 также возвращаются в формате проблем RFC 7807.
- Scalar и OpenAPI должны быть доступны не только в Development.
- Группы конечных точек модулей по умолчанию требуют авторизацию.
- Анонимные конечные точки должны быть явно ограничены health-check, служебной метаинформацией или сценариями аутентификации.

## Авторизация и владение данными

- Обработчики, которые читают или меняют пользовательские данные, используют `IUserContext`.
- Мутации проверяют владение до изменения агрегата.
- Чтение фильтрует данные по владельцу или возвращает ошибку доступа, если объект принадлежит другому пользователю.
- Будущие административные конечные точки должны использовать роли или политики.

## Роли

- Миграция Auth начально наполняет роли `admin` и `user`.
- Новый пользователь получает роль `user`.
- Названия ролей берутся из `AuthRoleNames`.

## Изображения и файлы

- Все изображения хранятся в MinIO.
- Добавление пользовательских изображений выполняется через backend: он проверяет владельца сущности, валидирует MIME-тип и размер файла, сохраняет объект в MinIO и пишет metadata-связь в БД.
- Бэкенд может проверять наличие объекта, сохранять объект или выдавать временную ссылку через `IObjectStorageService`.
- Фронт читает изображения напрямую из MinIO, API не должен проксировать обычную выдачу изображений без отдельной причины.
- В сущностях нельзя хранить произвольные URL как изображения. Нужно хранить metadata: `BucketName`, `ObjectKey`, тип привязки и дополнительные поля связи.
- Для рецептов ключи изображений строятся с префиксом `users/{ownerUserId:N}/recipes/{recipeId:N}/images/...`, чтобы будущие изображения шагов и производные файлы можно было добавить без смены модели хранения.
- У рецепта должна быть только одна активная обложка. Замена обложки помечает старую metadata удаленной и после сохранения очищает старый объект MinIO.
- Удаление изображения всегда выполняется через backend с проверкой владельца рецепта. Сначала меняется metadata, затем удаляется объект MinIO; сбой очистки объекта логируется, а не откатывает уже завершенное пользовательское удаление.

## Доступ к данным

- Repository и Unit of Work допустимы для командных сценариев, где нужно загрузить агрегат и применить доменные инварианты.
- Для запросов чтения не нужно по умолчанию гидрировать полную доменную сущность.
- Новые списковые и проекционные запросы лучше делать через интерфейс к DbContext/read model или через спецификации, возвращая DTO или проекцию.
