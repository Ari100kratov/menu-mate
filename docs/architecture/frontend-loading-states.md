# Состояния загрузки frontend

Skeleton должен повторять геометрию контента, который появится после завершения запроса. Универсальная последовательность из заголовка и прямоугольников не используется для feature-экранов: карточки рецептов, форма рецепта, календарь меню, список покупок, профиль и черновики импорта имеют собственные структурные компоненты загрузки.

Базовый `Skeleton` из `shared/ui` отвечает только за визуальный примитив. Компоновка хранится рядом с feature-компонентом, структуру которого она повторяет. Это позволяет изменять реальный экран и его состояние загрузки в одном архитектурном слое.

Правила поведения:

- уже загруженные данные не заменяются skeleton при фоновом обновлении; для этого используются `placeholderData` и компактный индикатор обновления;
- skeleton показывается при первой загрузке, когда полезного содержимого еще нет;
- статические элементы экрана остаются интерактивными и видимыми, если запрос загружает только вложенную область;
- количество элементов ограничивается типичным первым viewport, чтобы загрузка не создавала ложную высоту длинной страницы;
- корневой контейнер структурного skeleton имеет `role="status"` и доступное русское описание;
- при проверке на mobile и desktop размеры, сетка, отступы и breakpoint должны совпадать с итоговым компонентом.

Во время восстановления сессии показывается skeleton оболочки приложения: sidebar, верхняя панель, нижняя mobile-навигация и основная область. Feature-страница появляется только после подтверждения сессии.
