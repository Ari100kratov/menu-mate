# Создание рецепта из изображения

## Пользовательский поток

На странице рецептов кнопка добавления открывает меню с двумя сценариями: ручное создание и загрузка фото с ИИ. Сценарий импорта состоит из двух отдельных маршрутов:

- `/recipes/import` принимает от 1 до 8 скриншотов JPEG, PNG или WebP размером до 10 МБ каждый и до 40 МБ суммарно, а также показывает недавние черновики. Выбор файлов дополняет уже добавленный набор: каждый файл можно удалить до создания черновика, а изображение можно добавить из буфера обмена через `Ctrl+V`;
- `/recipes/import/:draftId` показывает все исходные скриншоты, предупреждения распознавания и обычную форму рецепта для проверки.

Изменения формы черновика сохраняются автоматически с задержкой. Рецепт создаётся только после явного нажатия «Создать рецепт». Исходные скриншоты не становятся обложкой автоматически. Черновик и исходники удаляются фоновым процессом через 7 дней после последнего изменения; срок настраивается через `RecipeImports:DraftRetentionDays`.

В формах создания, редактирования рецепта и проверки импортированного черновика рядом с загрузкой обложки доступна генерация фото блюда с ИИ. Генерация использует все текущие данные формы и возвращает квадратный JPEG `1024x1024` с низким качеством и сжатием, достаточным для карточки рецепта. Изображение остаётся локальным файлом формы и загружается как обычная обложка только при сохранении рецепта.

## Границы модулей

Модуль RecipeImports владеет черновиками, исходными изображениями и доказательствами распознавания. Он хранит данные в схеме PostgreSQL `imports`, а исходные изображения в закрытом бакете MinIO `imports`. Клиент получает только временную подписанную ссылку на исходник.

RecipeImports не записывает сущности Recipes напрямую. При подтверждении он вызывает команду создания Recipes с заранее выделенным идентификатором. Это делает повторное подтверждение идемпотентным при сбое между созданием рецепта и обновлением статуса черновика.

## Интеграция с OpenAI

Backend отправляет все изображения одним запросом в OpenAI-совместимый Chat Completions API и требует строгий JSON по схеме `CreateRecipeRequest` вместе с исходным распознанным текстом, исходной строкой каждого ингредиента и предупреждениями. Модель по умолчанию: `gpt-5.4-mini`. Порядковые номера в начале распознанных шагов удаляются детерминированно после ответа модели.

Явно указанное количество порций переносится без изменений. Если его нет, модель оценивает целое значение от 1 до 100 по типу блюда, общему количеству ингредиентов и обычному размеру взрослой порции. Значение `1` используется только для очевидно индивидуального блюда, а в предупреждения черновика добавляется сообщение о приблизительной оценке.

Допустимые единицы измерения задаются не данными PostgreSQL, а общим серверным словарём импорта. Из него одновременно строятся enum строгой JSON-схемы, русская памятка для модели и правила повторной проверки исходной строки. Поэтому добавление единицы нельзя применить только в промпте: словарь должен оставаться согласованным с доменным `MeasurementUnit` и frontend-справочником.

После ответа модели сервер повторно разбирает исходную строку каждого ингредиента и применяет правила без дополнительного обращения к ИИ:

- точная масса или объём приоритетнее количества в штуках или упаковках; вторичное количество сохраняется в комментарии;
- диапазон без точной массы хранится как нижняя граница в числовом поле и целиком сохраняется в комментарии, поэтому он не превращается в «по вкусу»;
- единица без числа означает одну единицу, например один пучок или одну щепотку;
- «стакан», «чашка», «зуб.», «зубчик» и другие формы сопоставляются с разными каноническими единицами;
- процент жирности или концентрации для соответствующих продуктов остаётся частью названия продукта, а не количеством или комментарием;
- объединённые моделью соль и перец разделяются на два ингредиента;
- одинаковые продукты, ошибочно созданные из одной исходной строки, схлопываются; разные продукты из общей строки, например соль и перец, сохраняются раздельно;
- «по вкусу» выбирается только при прямом указании в источнике либо при отсутствии и количества, и единицы.

Названия ингредиентов в `productName` извлекаются в словарной форме: именительный падеж, без количества и единицы измерения. Значимые характеристики продукта, например `33%` у сливок, остаются в названии. Это позволяет общему каталогу создать отдельный продукт `сливки 33%`, не смешивая его с обычными сливками.

Теги черновика нужны для дальнейшего поиска и структурирования библиотеки. Модель добавляет до 6 содержательных тегов: о кухне, способе приготовления, типе питания, поводе или практической потребности; основной ингредиент допустим, если его нет в названии. Тип блюда и приём пищи («завтрак», «обед», «суп», «выпечка» и аналоги), название блюда, слова «еда» и «рецепт», дубли и слишком общие слова в теги не попадают.

## Обложка из исходного изображения

Тот же vision-запрос может предложить одну фотографию готового блюда из исходных изображений. Модель возвращает индекс изображения, уверенность и нормализованную область фотографии в координатах `0–1000`. Фотографии ингредиентов, процесса и отдельных шагов, текст, интерфейс сайта, реклама и коллаж целиком не считаются подходящей обложкой. Если готовое блюдо нельзя определить уверенно, предложение отсутствует.

Сервер принимает предложение только при уверенности не ниже 70 и корректных координатах внутри существующего исходника. Некорректное предложение отбрасывается без ошибки импорта. Метаданные сохраняются в `EvidenceJson` как nullable-поле, поэтому отдельная миграция БД не нужна, а старые черновики остаются совместимыми.

Предложение никогда не применяется автоматически. В форме пользователь открывает квадратный редактор, перемещает изображение и меняет масштаб, после чего явно выбирает «Использовать как обложку». Исходник читается через авторизованный маршрут RecipeImports с проверкой владельца. Браузер вырезает выбранную область через Canvas в JPEG с качеством 0,88 и стороной не более 1024 пикселей без увеличения исходных пикселей.

До подтверждения рецепта вырезанный файл существует только в памяти браузера. При создании рецепта он загружается обычным механизмом Recipes как `Cover`. Отказ от предложения, ошибка чтения исходника или ошибка Canvas не блокируют редактирование: ручная загрузка и отдельная ИИ-генерация обложки остаются доступными.

Обложка создаётся через OpenAI Image API. Модель по умолчанию: `gpt-image-1-mini`; параметры изображения: качество `low`, размер `1024x1024`, формат JPEG, сжатие 75%.

Ключ распознавания задаётся через `OpenAI:ApiKey` или `OPENAI_API_KEY`; хранить его в репозитории или передавать во frontend запрещено. Ответы модели не сохраняются на стороне провайдера через параметр `StoredOutputEnabled = false`.

Для распознавания используется OpenAI-совместимый endpoint `chat/completions`: изображения передаются вместе с текстовой инструкцией, а ответ ограничен JSON-схемой черновика. Поэтому сценарий работает как с прямым OpenAI API, так и с AITunnel, который документирует этот endpoint для `gpt-5.4-mini` и входных изображений.

Чтобы включить AITunnel в Docker/Portainer, задайте:

```env
OPENAI_API_KEY=ваш_ключ_aitunnel
OPENAI_BASE_URL=https://api.aitunnel.ru/v1/
OPENAI_MODEL=gpt-5.4-mini
```

`OPENAI_BASE_URL` оставляют пустым для прямого OpenAI API. Те же `OPENAI_API_KEY` и `OPENAI_BASE_URL` применяются к генерации обложки через Image API. Значения не передаются во frontend и не должны попадать в репозиторий.

Доступные настройки:

- `OpenAI:ApiKey` / `OPENAI_API_KEY` — общий ключ распознавания рецепта и генерации обложки;
- `OpenAI:BaseUrl` / `OPENAI_BASE_URL` — необязательный базовый URL OpenAI-совместимого API для распознавания и генерации обложек;
- `OpenAI:Model` — модель распознавания рецепта;
- `OpenAI:ImageModel` — модель генерации обложки;
- `RecipeImports:DraftRetentionDays` — срок хранения черновика после последнего изменения;
- `RecipeImports:CleanupIntervalMinutes` — интервал фоновой очистки.

При прямом запуске API настройки видны в `src/MenuMate.Api/appsettings.Development.json`. При запуске через Aspire используются настройки из `src/MenuMate.AppHost/appsettings.Development.json`, после чего AppHost передаёт их API. Реальный ключ рекомендуется хранить в user secrets или переменной окружения, а поле `ApiKey` в репозитории оставлять пустым.

## API

Базовый маршрут: `/api/imports/recipe-drafts`.

- `POST /` создаёт черновик из multipart-коллекции `files`;
- `GET /` возвращает недавние черновики пользователя;
- `GET /{draftId}` возвращает черновик и временную ссылку на исходник;
- `GET /{draftId}/source-images/{sourceImageIndex}/content` авторизованно возвращает байты конкретного исходника для редактора обложки;
- `PUT /{draftId}` заменяет редактируемое содержимое;
- `POST /{draftId}/confirm` создаёт рецепт;
- `DELETE /{draftId}` удаляет черновик и все исходные файлы.

Отдельный защищённый маршрут `POST /api/recipe-images/generate-cover` принимает `CreateRecipeRequest` и возвращает сгенерированный JPEG.
