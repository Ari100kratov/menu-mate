# Модули

## Auth

Владеет пользователями, ролями, refresh tokens и выпуском JWT.

- регистрация, вход, профиль и logout;
- ротация opaque refresh token в `HttpOnly` cookie;
- access token только в ответе API и памяти frontend;
- роли `admin` и `user`;
- схема PostgreSQL `auth`.

## Recipes

Владеет рецептами, их авторством, видимостью, ревизиями, библиотекой пользователя и изображениями.

- CRUD личных рецептов и чтение публичных;
- неизменяемые ревизии при сохранении содержимого;
- библиотека, избранное и приватная копия чужого рецепта;
- ингредиенты со ссылками на Products, шаги, категория блюда, времена и теги;
- масштабирование ингредиентов без изменения исходника;
- обложка и по одному изображению для шага в MinIO;
- схема PostgreSQL `recipes`.

Подробное поведение ревизий, библиотеки, копий, меню и удаления описано в документе [«Версионирование, копии и жизненный цикл рецептов»](recipe-versioning.md).

## Products

Владеет общим каталогом продуктов для рецептов и покупок.

- идентичность задаётся парой нормализованного названия и категории;
- одинаковое название допустимо в разных категориях;
- поиск для автодополнения;
- создание отсутствующего продукта в пользовательском сценарии;
- схема PostgreSQL `products`.

## Tags

Владеет глобальным для всей системы справочником тегов, нормализацией названий, типом и статусом подтверждения. Теги являются дополнительной классификацией рецепта и не заменяют его основную категорию блюда. Данные хранятся в схеме `tags`.

- `tags.tags` хранит единственное каноническое название; уникальный индекс по нормализованному названию исключает дубли во всей системе;
- `recipes.recipe_revision_tags` хранит только пары `recipe_revision_id` и `tag_id`; название в Recipes не копируется, отдельной таблицы текущих тегов рецепта нет;
- актуальные теги рецепта определяются через `recipes.current_revision_id`, а связи прошлых ревизий сохраняют историю;
- при ручном создании и редактировании рецепта отсутствующие теги регистрируются как `User / Confirmed`;
- при подтверждении импорта из изображений отсутствующие теги регистрируются как `Suggested / Confirmed`, то есть считаются принятыми вместе с черновиком;
- публичный API пользователя не позволяет переименовывать, подтверждать, скрывать или удалять теги; такие операции зарезервированы для будущего административного интерфейса;
- миграция собирает старые теги рецептов и ревизий в глобальный справочник, повторно использует уже существующую запись по нормализованному имени и сразу создаёт связи по `tag_id`;
- поиск ограничивается 30 результатами и ранжируется так: точное совпадение, начало названия, отдельное слово, прочее вхождение подстроки; поиск по подстроке поддержан `pg_trgm` и GIN-индексом;
- frontend не запрашивает и не раскрывает весь справочник до начала ввода пользователем.

## MenuPlanning

Владеет единым непрерывным календарём питания пользователя.

- диапазон дат является параметром чтения, а не отдельным планом;
- настраиваемые приёмы пищи с названием и порядком;
- стартовые «Завтрак», «Обед» и «Ужин» при регистрации; миграция также добавляет их существующим пользователям, созданным вне обычного регистрационного потока;
- позиции по рецепту или со свободным текстом;
- позиция рецепта фиксирует `RecipeId` и неизменяемый `RecipeRevisionId`;
- схема PostgreSQL `menu_planning`.

## ShoppingLists

Владеет единственным актуальным списком покупок пользователя.

- ручное добавление, редактирование, удаление и отметка «куплено»;
- предпросмотр диапазона меню с изменением порций и исключением ингредиентов;
- полная замена списка только после явного подтверждения;
- объединение совместимых позиций по `ProductId` и детерминированные преобразования кг→г, л→мл;
- группировка по категории продукта и текстовая версия для копирования;
- схема PostgreSQL `shopping_lists`.

## RecipeImports

Владеет черновиками импорта из изображений и исходными файлами.

- от 1 до 8 JPEG, PNG или WebP на черновик;
- распознавание через OpenAI-совместимый Chat Completions API;
- редактирование и обязательное подтверждение результата;
- идемпотентное создание рецепта с заранее выделенным идентификатором;
- фоновая очистка просроченных черновиков;
- схема PostgreSQL `imports` и закрытый бакет MinIO `imports`.

## Межмодульные связи

- Recipes использует каталог Products и идентификаторы глобальных Tags, но остаётся владельцем состава и истории ревизий рецепта.
- MenuPlanning хранит ссылку на конкретную ревизию Recipes и snapshot отображаемого названия.
- ShoppingLists читает зафиксированные ревизии рецептов для расчёта покупок и объединяет строки по `ProductId`.
- RecipeImports создаёт Recipes через application-команду, не изменяя таблицы рецептов напрямую.
- Auth предоставляет `IUserContext`; каждый пользовательский модуль самостоятельно проверяет владение своими данными.
