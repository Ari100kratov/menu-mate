# Модули

## Auth

Владеет пользователями, ролями, refresh tokens и выпуском JWT.

Сейчас реализовано:

- регистрация и вход;
- refresh token rotation;
- конечная точка профиля текущего пользователя;
- logout через отзыв refresh tokens;
- схема PostgreSQL `auth`.

## Рецепты

Владеет рецептами, ссылками на продукты каталога, количеством ингредиентов, шагами приготовления, категорией блюда, временем, флагом избранного, URL источника и snapshot тегов рецепта.

Сейчас реализовано:

- `Recipe`
- `RecipeIngredient`
- `IngredientQuantity`
- количество ингредиента хранится без отдельного типа точности: числовое значение задается через `amount + unit`, а «по вкусу» — через `unit = ToTaste` без `amount`
- `RecipeTag`
- persistence в схеме `recipes`
- Recipes CRUD API
- масштабирование ингредиентов без изменения исходного рецепта
- владение через `UserId`

## Продукты

Владеет общим каталогом продуктов для рецептов и списков покупок.

Сейчас реализовано:

- уникальная пара нормализованного названия и категории продукта; одинаковое название допустимо в разных категориях;
- категория продукта для группировки покупок;
- поиск для автодополнения;
- persistence в схеме `products`;
- разрешение нового названия в существующий или новый продукт.

## Tags

Владеет жизненным циклом тегов и нормализацией названий.

Сейчас реализовано:

- `Tag`
- `TagName`
- тип тега и статус подтверждения
- persistence в схеме `tags`
- Tags API

## Планирование меню

Владеет календарем питания пользователя, настраиваемыми приемами пищи и позициями меню.

Сейчас реализовано:

- единый календарь пользователя без отдельных именованных планов;
  - выбор диапазона дат как состояние запроса, а не persistence-сущность;
  - настраиваемые приемы пищи с пользовательским названием и порядком;
  - создание обычных приемов пищи «Завтрак», «Обед» и «Ужин» при регистрации пользователя;
- позиции меню на основе рецептов с snapshot названия рецепта;
- свободный текст без автоматического создания рецепта;
- количество порций на пункт меню;
- persistence в схеме `menu_planning`;
- MenuPlanning API;
- владение через `UserId`;
- связь с Recipes через `RecipeId` и immutable `RecipeRevisionId`.

## ShoppingLists

Владеет единственным актуальным списком покупок пользователя, его ручными позициями и заменой из меню.

Сейчас реализовано:

- перевод кг в г;
- перевод л в мл;
- осторожное объединение совместимых позиций;
- сохранение позиций «по вкусу» отдельными строками;
- группировка по категориям;
- persistence в схеме `shopping_lists`;
- ShoppingLists API;
- один список на владельца с уникальным индексом по `OwnerUserId`;
- предпросмотр диапазона календаря меню с выбором ингредиентов и изменением порций;
- полная замена текущего списка после явного подтверждения;
- ручное добавление, редактирование, удаление и отметки позиций;
- связь позиций с общим каталогом продуктов;
- объединение совместимых позиций по идентификатору продукта;
- текстовая версия списка для копирования.

## Повторное использование рецептов между модулями

- Recipes владеет авторством, видимостью, пользовательской библиотекой, неизменяемыми ревизиями и происхождением копий.
- MenuPlanning хранит `RecipeId` и `RecipeRevisionId` для позиций на основе рецепта.
- ShoppingLists читает ингредиенты зафиксированной ревизии и не использует изменяемые текущие ингредиенты рецепта.
- Названия рецептов являются отображаемыми данными и намеренно не уникальны.

## Imports

Планируемый модуль для импорта из URL и изображений. Он должен хранить черновики, исходные доказательства извлечения и предложенные теги. Он не должен напрямую записывать данные в Recipes без подтверждения пользователя.

## Files

Планируемый модуль для загрузки и проверки файлов. Все изображения должны храниться в MinIO.

## Family Spaces

Планируемое расширение модели владения для общих книг рецептов, планов меню и списков покупок.
