# Архитектура фронтенда

Фронтенд находится в `src/MenuMate.Web`. Это Vite + React + TypeScript SPA с mobile-first интерфейсом и ручным feature-слоем поверх типизированного API-клиента.

## Структура

```text
src/MenuMate.Web/
  scripts/
    generate-api.mjs
  src/
    app/
    pages/
    features/
    shared/
      api/
        generated/
      auth/
      config/
      lib/
      ui/
```

`shared/api/generated` создается командой `pnpm api:generate` и вручную не редактируется. Feature-слой пишет query hooks сам, чтобы не смешивать контракты backend с UX-состояниями экранов.

Page-компоненты являются route orchestration: загрузка данных верхнего уровня, выбор активной сущности, `PageHeader`, обработка пустых и ошибочных состояний. Формы, карточки, рабочие области, local UI state, form hooks, Zod-валидация и form-to-DTO маппинг выносятся в `features/*/ui` и `features/*/model`. Практический ориентир для ручного кода — не выращивать файлы за счет нескольких сценариев сразу; если файл приближается к двум сотням строк, сначала выделяется подкомпонент, хук или model-helper. Сгенерированные OpenAPI-типы являются исключением.

## API

OpenAPI backend должен быть пригоден для генерации: у endpoint-ов указываются success-схемы, `ProblemDetails`, авторизация и возможные auth-ответы. Типы генерируются через `openapi-typescript`, клиент строится на `openapi-fetch`.

Локально фронт ходит в API через same-origin `/api`, а Vite proxy направляет запросы в backend. Это важно для cookie-flow: refresh-token хранится в `HttpOnly` cookie и не попадает в JavaScript.

Генератор OpenAPI запускает backend локально и передает `MENUMATE_DATA_PROTECTION_KEYS_PATH`, чтобы tooling-сценарий использовал изолированные DataProtection-ключи и не зависел от пользовательского key ring Windows.

Standalone Vite-запуск по умолчанию проксирует `/api` и `/openapi` на `http://127.0.0.1:5020`, потому что это порт `MenuMate.Api` из `launchSettings.json`. Aspire может переопределить адрес через `VITE_API_PROXY_TARGET`.

## Авторизация

Access-token хранится только в памяти Zustand-store. Refresh-token живет в `HttpOnly` cookie, выставляется backend-ом и используется только запросом `/api/auth/refresh`.

Если API возвращает `401`, общий fetch-клиент пытается один раз обновить access-token и повторить исходный запрос. Если refresh не удался, клиент очищает локальное состояние сессии.

Кастомный fetch-wrapper обязан сохранять headers исходного `Request`, созданного `openapi-fetch`. Иначе можно случайно потерять `Content-Type: application/json` при добавлении `Authorization`.

После явного logout или проваленного refresh клиент блокирует дальнейшие refresh-попытки до нового login/register. `/api/auth/logout` не запускает автоматический refresh при `401`, потому что выход не должен превращаться в восстановление сессии.

## Пользовательские предпочтения

Локальные UI-предпочтения хранятся в `shared/config/user-preferences.store.ts` через Zustand persist. Этот слой допустим только для клиентских дефолтов, которые не являются источником истины backend: например, дефолтная единица и категория для ручной позиции покупок. Если настройка должна синхронизироваться между устройствами, участвовать в правах доступа или влиять на доменные правила, для нее нужен отдельный backend-контракт.

Справочные значения единиц и категорий, которые используются несколькими feature-областями, лежат в `shared/config/shopping-taxonomy.ts`. Feature-компоненты не должны дублировать эти списки локально.

## Формы

Формы строятся через TanStack Form и Zod. Для полей используется shadcn-подход с отдельными `Field`, `FieldLabel`, `FieldError`, чтобы валидация, доступность и верстка были одинаковыми во всех feature-формах. Схемы формы лежат в model-файлах фичи, а form hooks живут рядом с UI формы; route pages не должны содержать правила валидации. Ручные пункты меню и покупок проверяются на клиенте до запроса: нельзя отправлять пустой продукт, пустой пункт меню без рецепта/текста, некорректную дату плана или невалидное количество.

TypeScript зафиксирован на ветке 5.x, потому что `openapi-typescript` 7.x имеет peer dependency `typescript ^5.x`.

Первый полноценный feature-срез — `features/recipes`. В нем API-вызовы лежат в `features/recipes/api`, query/mutation hooks оборачивают TanStack Query, а форма создания и редактирования построена на TanStack Form с массивами ингредиентов и шагов. Для CRUD-форм detail-response backend обязан возвращать все редактируемые поля. Если поле принимается в `Create/Update` запросе, но не возвращается в detail-response, форму нельзя считать безопасной для редактирования: пользователь может открыть сущность и сохранить ее с потерей данных.

Списки и проекции не обязаны загружать полный detail-контракт. Для списка рецептов используется `RecipeListItemResponse`, а для формы редактирования — `RecipeResponse`.

Поток `диапазон календаря меню -> предпросмотр -> список покупок` разделен на два feature-слоя: `features/menu-planning` и `features/shopping-lists`. Экран меню передает выбранные `startDate` и `endDate` на `/shopping/preview`. После выбора ингредиентов и подтверждения backend полностью заменяет единственный список покупок пользователя. Глобальный Zustand-state не нужен: TanStack Query хранит серверный список, а выбор предпросмотра является локальным состоянием страницы.

Страницы `MenuPage` и `ShoppingPage` остаются рабочими mobile-first экранами. `MenuPage` группирует дни и приемы пищи вертикально, а `ShoppingPage` показывает единственный список, сгруппированный по категориям. Ручные формы отправляют backend DTO без хранения дублирующего глобального состояния.

`MenuPage` показывает единый календарь питания за выбранный диапазон. Добавление и редактирование позиции используют реальные backend-мутации: `POST /api/menu-calendar/items` и `PUT /api/menu-calendar/items/{itemId}`. UI не должен подменять редактирование удалением и повторным добавлением, потому что это ломает идентичность позиции и будущие связи с заметками, уведомлениями или аналитикой.

`ShoppingPage` показывает один список без фильтров, режима магазина и статуса «есть дома». Позиция имеет одну отметку «куплено». Добавление и редактирование открываются в адаптивном диалоге, удаление и изменение отметки сразу синхронизируются через `/api/shopping-list/items`. На телефоне ручное добавление доступно через одну плавающую кнопку.

Детальная страница рецепта открывает единый календарь в режиме размещения выбранного рецепта. Пользователь может последовательно нажать «Добавить сюда» в нескольких датах и приемах пищи, после чего завершить режим одной кнопкой. Выбранный рецепт передается через query string, а успешные добавления обновляют календарный кэш TanStack Query. Date helpers и состояние диапазона лежат в `features/menu-planning/model`, а не дублируются в page-компонентах.

## UI

Интерфейс mobile-first: на телефоне основные разделы доступны через нижнюю навигацию, на desktop используется sidebar. Базовые разделы: рецепты, меню, покупки, профиль.

Теги не являются отдельным основным разделом MVP. Пользователь работает с ними внутри сценариев рецептов: вводит теги в форме, видит их в карточках и использует как фильтр. Отдельный справочник тегов и модерация через Tags API могут появиться позже как админский или настроечный сценарий, но такой экран не должен занимать место в основной мобильной навигации без отдельного продуктового основания.

Inline-работа с тегами живет в `features/tags`: API/query слой обращается к `Tags API`, модель `tag-values` отвечает за парсинг, нормализацию и сериализацию значений, а `TagPicker` используется формой рецепта. `features/tags` не должен импортировать `features/recipes`, потому что справочник тегов является самостоятельной feature-областью. Форма рецепта хранит значение поля как строку для TanStack Form, но перед отправкой преобразует его в `string[]` backend-контракта.

Новый тег, созданный через `TagPicker`, сначала сохраняется в справочнике через `POST /api/tags`, а затем добавляется в форму рецепта как выбранный chip. Это делает автодополнение полезным для следующих рецептов и не смешивает справочник тегов с агрегатом рецепта.

Компоненты добавляются по shadcn/ui-подходу как локальные файлы в `shared/ui`. Темизация построена на CSS tokens и `next-themes`.

Для повторяемых mobile-first блоков используются общие компоненты `PageSection` и `MobileStickyActions` из `shared/ui/page`. `PageSection` нужен для компактных рабочих блоков с заголовком, описанием и локальными действиями. `MobileStickyActions` используется только для основных действий текущего экрана на телефоне и не должен заменять desktop actions в header/sidebar.

Карточки списков должны сохранять стабильную высоту и понятное состояние без медиа. Если у рецепта нет обложки, UI показывает нейтральный placeholder, а не схлопывает карточку. Действия карточки должны оставаться доступными на телефоне без hover-сценариев.

Редактирование изображений рецепта живет на странице редактирования рецепта. Обложка загружается с `scope=Cover`, изображения шагов — с `scope=Step` и `stepNumber`; frontend не строит object key и не пишет произвольные URL.

План меню на телефоне использует горизонтальный выбор дня и показывает один активный день, чтобы недельная сетка не превращалась в длинную простыню. Быстрые действия ячейки только предзаполняют форму или вызывают существующие `POST/PUT` мутации: копирование создает новый пункт, перенос меняет дату существующего пункта.

## Ошибки и загрузка

Ошибки API приводятся к RFC7807 `ProblemDetails` и показываются через общий `ErrorAlert`. Долгие операции получают явные loading/skeleton-состояния, а пустые разделы используют общую `EmptyState`.

Общий unwrap API-клиента обязан проверять не только `error` из `openapi-fetch`, но и сам `response.ok`. Если endpoint должен вернуть JSON, пустой успешный ответ считается ошибкой контракта.

## Изображения

Фронтенд читает изображения напрямую из MinIO по `readUrl`, полученному от API. Добавление изображений идет через backend, потому что backend проверяет владельца рецепта, валидирует файл и создает metadata-связь в БД.

Frontend не добавляет собственные cache-busting параметры к `readUrl`. Backend переиспользует одну подписанную ссылку большую часть срока ее действия, поэтому стандартного HTTP-кеша браузера достаточно: повторная отрисовка `<img>` с тем же URL использует сохраненный ответ. Срок свежести ответа на уровне reverse proxy или браузера не должен превышать оставшийся срок действия подписи.

В MVP UI загружает, заменяет и удаляет обложку рецепта на странице редактирования. Загрузка идет через multipart endpoint `POST /api/recipes/{recipeId}/images`, удаление — через `DELETE /api/recipes/{recipeId}/images/{imageId}`. Тот же upload-контракт уже поддерживает `scope=Step` и `stepNumber`, чтобы позже добавить изображения конкретных шагов без смены API-подхода.

Список рецептов и read-only страница рецепта используют `coverImage`/`images` из API и показывают изображение только по `readUrl`. Если ссылки нет, UI показывает нейтральное состояние без попыток строить URL самостоятельно.

Фронт не должен сохранять произвольные URL как изображения сущностей. Для рецептов используются metadata-поля `bucketName`, `objectKey`, `scope`, `stepNumber` и `readUrl`. `SourceUrl` рецепта остается внешней ссылкой на страницу-источник, а не media-сущностью.
