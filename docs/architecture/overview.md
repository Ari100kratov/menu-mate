# Обзор архитектуры

MenuMate стартует как модульный монолит, а не как набор микросервисов.

Так локальная разработка, транзакции и deployment остаются простыми, но границы модулей уже видны в проектах, contracts, DbContext и схемах БД. Модуль можно вынести в отдельный сервис позже, если появится реальное операционное давление.

## Принципы

- Доменная логика не живет в конечных точках API и фронтенд-компонентах.
- Модули общаются через публичные контракты, прикладные сценарии или события.
- Stateful модуль владеет своей persistence-моделью.
- Межмодульные связи выражаются явными типами идентификаторов, например `UserId` и `RecipeId`.
- Данные другого модуля передаются как snapshot/contract, а не через прямой доступ к чужим сущностям.
- Документация и тесты являются частью продукта.

## Форма бэкенда

```text
MenuMate.Api
MenuMate.Contracts
MenuMate.SharedKernel
MenuMate.Common.*
Modules/<ModuleName>/
  MenuMate.Modules.<ModuleName>.Domain
  MenuMate.Modules.<ModuleName>.Application
  MenuMate.Modules.<ModuleName>.Infrastructure
  MenuMate.Modules.<ModuleName>.Presentation
```

## Владение данными

Основное направление — одна PostgreSQL база с отдельной схемой на stateful модуль. Это дает понятное владение данными без сложности распределенной системы на этапе MVP.

## Импорт рецептов

Импорт из URL или изображений сначала создает черновик. Рецепт нельзя сохранять без явного подтверждения пользователя.
