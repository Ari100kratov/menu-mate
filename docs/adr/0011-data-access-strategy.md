# 0011 Стратегия доступа к данным

## Статус

Принято.

## Контекст

Repository + Unit of Work удобны для командных сценариев, где нужно загрузить агрегат и выполнить доменные инварианты. Но для конечных точек чтения такой подход часто заставляет читать полную доменную сущность, хотя UI нужен небольшой DTO или список.

## Решение

Используем гибридный подход:

- командная сторона может использовать repository и unit of work для агрегатов;
- сторона чтения не должна по умолчанию гидрировать доменную модель;
- новые списковые и проекционные запросы должны идти через DbContext-интерфейс/read model или спецификации;
- прикладной слой должен зависеть от интерфейса чтения, а не от конкретного EF DbContext;
- прямой доступ к таблицам другого модуля запрещен, даже если база физически одна.

## Последствия

- Доменные инварианты остаются в агрегатах.
- Конечные точки чтения становятся дешевле и проще.
- В будущем можно выделять read models или отдельные БД без переписывания публичных contracts.
