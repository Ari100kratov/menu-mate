# 0009 Правила качества и безопасности backend-разработки

## Статус

Принято.

## Контекст

Бэкенд уже содержит авторизацию, модульное хранение данных, запуск через Aspire, миграции EF Core, Scalar и MinIO. Такие сквозные правила должны быть записаны явно, иначе будущие изменения могут незаметно ослабить тестируемость, безопасность и консистентность API.

## Решение

Все backend-изменения должны следовать правилам из [правил backend-разработки](../engineering/backend-guardrails.md).

Ключевые инварианты:

- persisted ID создаются через GUID version 7;
- время приходит через `TimeProvider`;
- warnings, code style и analyzers валят сборку;
- ошибки возвращаются в формате проблем RFC 7807;
- защищенные конечные точки требуют авторизацию;
- пользовательские данные проверяются через `IUserContext`;
- межмодульные связи выражаются явными типами `UserId`, `RecipeId` и аналогами;
- EF Core конфигурации сущностей вынесены в отдельные классы;
- Scalar и OpenAPI доступны в Development, а в production требуют явного диагностического флага;
- изображения хранятся в MinIO.

## Последствия

- Сборка намеренно строже дефолтной SDK-конфигурации.
- Подавления анализаторов допустимы только для понятных false positive.
- Новый модуль должен сразу получать структуру каталогов, EF-конфигурации, правила авторизации и явные cross-module identifiers.
