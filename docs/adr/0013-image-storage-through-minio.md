# 0013 Изображения через MinIO

## Статус

Принято.

## Контекст

В проекте появляются изображения рецептов, а позже могут появиться изображения конкретных шагов приготовления, аватары, изображения импортированных источников и другие media-типы. Нельзя хранить бинарные файлы в PostgreSQL и нельзя привязывать доменные сущности к произвольным внешним URL как к основному хранилищу изображений.

Поле `SourceUrl` у рецепта не является изображением. Это внешняя ссылка на страницу-источник рецепта, поэтому она остается абсолютным URL и не заменяет media-хранилище.

## Решение

Все пользовательские изображения хранятся в MinIO. Backend принимает новые изображения, проверяет права владельца, валидирует тип и размер файла, сохраняет объект в MinIO и только после этого пишет метаданные в базу данных модуля-владельца.

Для MVP используется один бакет пользовательских изображений:

- `images`

Разделение по предметным областям делается через ключ объекта, а не через отдельный бакет на каждую сущность. Бакеты являются инфраструктурной границей, а не доменной моделью.

Структура ключей:

```text
users/{ownerUserId:N}/recipes/{recipeId:N}/images/cover/{imageId:N}.{extension}
users/{ownerUserId:N}/recipes/{recipeId:N}/images/steps/{stepNumber}/{imageId:N}.{extension}
```

В PostgreSQL хранится только metadata-запись: `imageId`, `ownerUserId`, `recipeId`, область привязки (`Cover` или `Step`), номер шага, имя бакета, ключ объекта, MIME-тип, размер, исходное имя файла, alt-текст и дата создания.

API может возвращать временную или публичную ссылку чтения `readUrl`, но само чтение изображения выполняет фронтенд напрямую из MinIO. API не проксирует обычную выдачу изображений.

Для рецепта может быть только одна активная обложка. При загрузке новой обложки backend помечает предыдущие активные cover-записи удаленными в той же транзакции metadata, сохраняет новую запись, а затем пытается удалить старые объекты из MinIO.

Удаление изображения выполняется через backend. Сначала metadata-запись помечается удаленной, затем объект удаляется из MinIO. Если очистка объекта не удалась после успешного изменения metadata, пользовательский сценарий считается завершенным, а ошибка логируется для последующей фоновой или ручной очистки.

## Последствия

Backend остается единственным местом добавления изображений, потому что только он может надежно проверить владельца рецепта и привязать объект к metadata-записи.

Удаление или замена изображения должны работать в две фазы: сначала изменить metadata в БД, затем удалить объект из MinIO или поставить его на фоновую очистку. Если запись metadata не сохранилась после успешной загрузки объекта, backend пытается удалить загруженный объект.

Если появятся производные изображения, например миниатюры, они должны храниться либо в том же бакете с отдельным префиксом `derivatives`, либо в отдельном бакете `image-derivatives` после отдельного ADR. Исходники пользовательских изображений остаются в `images`.

Контракты сущностей не должны хранить произвольные URL изображений. Для связи с изображением используется явная metadata-сущность с `BucketName` и `ObjectKey`.
